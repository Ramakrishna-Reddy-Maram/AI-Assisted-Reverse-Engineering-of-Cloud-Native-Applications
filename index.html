01

The bottleneck

Reverse engineering recovers meaning from compiled code with no source attached. The tools — Ghidra, IDA Pro, Radare2 — are good. The bottleneck has always been human attention.

Why it doesn't scale

  • A medium-complexity function takes roughly 25 minutes to read, annotate, and document by hand (Mantovani et al., 2022).
  • Cloud-native systems split logic across dozens of separately-compiled microservices — the review volume scales with service count, not app complexity.
  • Ten microservices at 15 functions each is 62 hours of reading before a single finding gets written down.

What changes with an LLM in the loop

  • Models trained on code can read Ghidra's decompiled C and produce a usable explanation in under two seconds.
  • The explanation isn't always right on obfuscated code or across function boundaries — but it replaces a blank page with a starting point.
  • For triage work — ranking functions by apparent risk before a human reviews them — that's the difference that matters.
02

How start.py works

One file. No external config, no database, no separate setup step — drop it on any machine running Python 3 and it handles everything from dependency installation to the final HTML report.

STAGE 0

Setup

Installs its own dependencies via pip, resolves paths with Path.home() so it runs identically on Linux, macOS, Windows, and WSL.

STAGE 1

Extract

Regex-based function extraction with brace-depth tracking pulls complete function bodies from the C source, with a line-by-line fallback for non-standard formatting.

STAGE 2

Analyze

Each function goes to Groq's LLaMA 3-70B at temperature 0.1, returning a fixed JSON schema: summary, component type, security issues, logic reconstruction, complexity.

STAGE 3–4

Report

Per-function speedup is computed against Mantovani et al.'s manual baselines, then rendered into an HTML report with Chart.js visualizations.

03

What it found

The target was a purpose-built six-function authentication microservice with deliberately seeded vulnerabilities — the kind that show up in real production code. The model flagged 16 distinct issues across five functions with zero false positives.

authenticate_user()
int authenticate_user(char *user, char *pass) {
  unsigned long h = simple_hash(pass);
  if (strcmp(stored_hash, h) == 0) {
    char *tok = gen_token();
    return AUTH_OK;
  }
  return AUTH_FAIL;
}
Security · Medium complexity 4 issues
  • Non-cryptographic hash. DJB2 + XOR mixing is a hash-table function, not a password function — fast by design, which is the opposite of what credential storage needs.
  • Timing leak. strcmp returns on first-byte mismatch, letting an attacker infer how many characters of a guess matched.
  • No rate limiting on login attempts.
  • Predictable tokens. rand() seeded with time(NULL).
register_user()
int register_user(char *user, int role) {
  /* role: unvalidated int */
  users[g_user_count].role = role;
  g_user_count++;
  strncpy(users[g_user_count].name, user, 32);
  return REG_OK;
}
Authentication · Medium complexity 4 issues
  • Privilege escalation. Role is an unvalidated integer — passing 1 grants admin.
  • No password complexity enforcement.
  • Race condition. Global user count incremented with no synchronization in a multithreaded context.
  • strncpy with a fixed length but no interior null-byte check.
simple_hash()
unsigned long simple_hash(char *s) {
  unsigned long h = 5381;
  while (*s) {
    h = ((h << 5) + h) ^ *s++;
  }
  return h; /* 64-bit output */
}
Security · Low complexity 3 issues
  • DJB2 is not cryptographically secure.
  • XOR mixing with hardcoded constants adds no real entropy.
  • Effective entropy is capped by the 64-bit DJB2 output space.

What it missed

A manual audit of the binary turned up 18 distinct issues. The model found 16 — both misses share the same root cause.

Both missed issues need cross-function context

  • An integer overflow risk in the user-count comparison, which depends on the relationship between g_user_count and MAX_USERS — defined in a different function.
  • An uninitialized memory read inside validate_token that only surfaces under unusual calling conditions from elsewhere in the codebase.

Where it held up

  • Flagged DJB2 as non-cryptographic with no hint about which algorithm was in use.
  • Caught the strcmp timing pattern independently in two separate functions.
  • All eight component-type classifications were correct; valid JSON returned on every call, no retries needed.
  • Ran unmodified on three different machines, and on a second 14-function test application, with zero configuration changes.
Triage, not replacement

The tool's limitation is structural: it reads functions in isolation. Both of the issues it missed required reasoning across function boundaries — call graphs are the natural next step. What it does well is exactly what cloud-native review needs most: a first pass that ranks risk and gives an analyst a starting point instead of a blank page.
